# JeJe WebOS NAS 部署指南 (v2.5.35)

本指南适用于在 **飞牛NAS (FnOS)**、群晖 (Synology)、威联通 (QNAP) 等 NAS 设备上通过 Docker 部署 JeJe WebOS。

---

## 📦 部署包内容

```
deployment/
├── jeje_webos_v2.5.35.tar    # Docker 镜像文件 (约 800MB)
├── docker-compose.yml       # Docker 服务编排配置
├── env_docker               # 环境变量配置文件
├── storage/                 # 数据存储目录
│   └── modules/
│       ├── ai/ai_models/    # AI 模型存放位置
│       ├── ocr/ocr_models/  # OCR 模型存放位置
│       └── map/map_tiles/   # 离线地图瓦片存放位置
└── 使用说明.md              # 本文档
```

---

## 🚀 快速部署 (5步完成)

### 步骤 1: 上传部署包

在 NAS 文件管理器中创建一个目录（如 `docker/jeje_webos`），将以下内容上传：

- [x] `jeje_webos_v2.5.35.tar` - 镜像文件
- [x] `docker-compose.yml` - 编排配置
- [x] `env_docker` - 环境变量
- [x] `storage/` 文件夹 - **务必完整上传**

### 步骤 2: SSH 连接 NAS

1. **开启 SSH**: 登录 NAS 网页端 → 设置 → SSH → 启用
2. **连接命令**:
   ```bash
   ssh 您的用户名@NAS_IP地址 -p 22
   # 示例: ssh admin@192.168.1.100 -p 22
   ```

### 步骤 3: 导入 Docker 镜像

```bash
# 进入部署目录（以飞牛NAS为例，路径可能是 /vol1/1000/docker/jeje_webos）
cd /vol1
find . -name "jeje_webos_v2.5.35.tar"   # 查找实际路径
cd /vol1/xxx/docker/jeje_webos          # 进入找到的目录

# 设置权限（重要！）
chmod -R 777 .

# 导入镜像（需要几分钟）
sudo docker load -i jeje_webos_v2.5.35.tar

# 验证导入成功
docker images | grep jeje
# 应显示: jeje_webos   v2.5.35   xxx   xxx   约800MB
```

### 步骤 4: 修改配置 (可选)

编辑 `env_docker` 文件，按需修改以下配置：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `APP_PORT` | 应用访问端口 | 9000 |
| `ADMIN_PASSWORD` | 管理员密码 | Admin@123 |
| `JWT_SECRET` | JWT 密钥 (建议修改) | 随机字符串 |
| `DB_PASSWORD` | 数据库密码 | jeje_123456 |
| `MYSQL_ROOT_PASSWORD` | MySQL root 密码 | root_123456 |

> 💡 **安全提示**: 生产环境强烈建议修改所有密码和密钥配置！

### 步骤 5: 启动服务

```bash
# 确保在 docker-compose.yml 所在目录
chmod -R 777 .

# 启动所有服务
sudo docker compose up -d

# 查看启动日志
sudo docker compose logs -f app
```

> 📝 如提示 `docker-compose: command not found`，请使用 `docker compose`（中间有空格）

### 🎉 访问应用

启动完成后，在浏览器访问：`http://您的NAS_IP:9000`

**默认账号**:
- 用户名: `admin`
- 密码: `Admin@123`（或您在 `env_docker` 中配置的密码）

## 🤖 AI 模型部署 (强烈推荐)

AI 助手功能需要本地大语言模型支持。

### 推荐模型

| 模型 | 规格 | 下载链接 |
|------|------|----------|
| **Qwen2.5-Coder-7B** (推荐) | 4.4GB, Q4量化 | [HuggingFace](https://huggingface.co/Qwen/Qwen2.5-Coder-7B-Instruct-GGUF/resolve/main/qwen2.5-coder-7b-instruct-q4_k_m.gguf) / [ModelScope](https://modelscope.cn/models/qwen/Qwen2.5-Coder-7B-Instruct-GGUF/files) |
| Qwen2.5-1.5B | 1.2GB, 轻量版 | [ModelScope](https://modelscope.cn/models/qwen/Qwen2.5-1.5B-Instruct-GGUF/files) |

### 部署步骤

1. 下载 `.gguf` 模型文件
2. 上传到 NAS 的 `storage/modules/ai/ai_models/` 目录
3. 重启应用服务: `sudo docker compose restart app`

> ⚠️ **注意**: 7B 模型需要约 8GB 内存，请确保 NAS 内存充足。内存不足时建议使用 1.5B 轻量模型。

## 📷 OCR 模型部署 (可选)

图文识别功能首次使用时会自动从网络下载模型。如 NAS 网络受限，可手动部署：

### 所需文件

| 文件名 | 用途 |
|--------|------|
| `ch_PP-OCRv4_det_infer.onnx` | 文字检测模型 |
| `ch_PP-OCRv4_rec_infer.onnx` | 文字识别模型 |
| `ch_ppocr_mobile_v2.0_cls_infer.onnx` | 方向分类模型 |

### 部署步骤

1. 参照 `storage/modules/ocr/ocr_models/` 目录中的说明文件下载模型
2. 将 3 个 `.onnx` 文件放入该目录
3. 重启应用即可使用离线 OCR

## 🗺️ 离线地图部署 (可选)

支持加载离线地图瓦片，适合内网环境使用。

### 部署步骤

1. 准备 XYZ 格式瓦片包（目录结构: `{z}/{x}/{y}.png`）
2. 将整个瓦片文件夹放入 `storage/modules/map/map_tiles/` 目录
3. 文件夹名称即为地图图层名称（建议使用英文）
4. 刷新网页，在地图图层列表中选择使用

## 📁 数据持久化说明

本部署方案采用**本地目录映射**，所有数据直接存储在 NAS 文件系统中：

| 数据类型 | 存储位置 |
|----------|----------|
| 用户上传文件 | `./storage/` |
| AI 模型 | `./storage/modules/ai/ai_models/` |
| OCR 模型 | `./storage/modules/ocr/ocr_models/` |
| 离线地图 | `./storage/modules/map/map_tiles/` |
| MySQL 数据 | Docker Volume: `jeje-mysql-data` |
| Redis 缓存 | Docker Volume: `jeje-redis-data` |

**备份建议**: 定期备份 `storage/` 目录和导出 MySQL 数据即可完整保留所有用户数据。

## 🔧 故障排除

### 问题 1: AI/OCR 模型无法加载

**症状**: 日志显示 `未找到可用的本地模型`

**诊断方法**:
```bash
# 进入容器检查
sudo docker exec -it jeje-webos-app /bin/bash

# 检查模型目录
ls -la /app/storage/modules/ai/ai_models/
ls -la /app/storage/modules/ocr/ocr_models/
```

**解决方案**:

```bash
# 1. 确保目录权限正确
chmod -R 777 ./storage

# 2. 重启服务
sudo docker compose restart app

# 3. 如果仍不生效，强制重建容器
sudo docker compose down
sudo docker compose up -d --force-recreate
```

### 问题 2: 浏览器缓存导致的 404 错误

**症状**: 页面显示异常或资源加载失败

**解决方案**:
1. 按 `Ctrl + Shift + Delete` 清除浏览器缓存
2. 按 `F12` → Application → Service Workers → Unregister
3. 按 `Ctrl + F5` 强制刷新

### 问题 3: 容器启动失败

**症状**: `docker compose up` 后容器反复重启

**诊断方法**:
```bash
# 查看容器日志
sudo docker compose logs app
sudo docker compose logs mysql
```

**常见原因及解决方案**:
- **端口冲突**: 修改 `env_docker` 中的 `APP_PORT`
- **MySQL 初始化慢**: 首次启动需等待 2-3 分钟，属正常现象
- **权限不足**: 执行 `chmod -R 777 .` 后重试

### 问题 4: 内存不足导致 AI 功能异常

**症状**: AI 对话时服务崩溃或响应超时

**解决方案**:
- 使用更小的模型（如 1.5B 版本）
- 在 `docker-compose.yml` 中调整 `shm_size` 值
- 关闭不使用的模块以释放内存

## 🔄 版本升级指南

### 升级步骤

```bash
# 1. 备份数据（重要！）
cp -r ./storage ./storage_backup_$(date +%Y%m%d)

# 2. 停止旧服务
sudo docker compose down

# 3. 上传并导入新版本镜像
sudo docker load -i jeje_webos_v新版本.tar

# 4. 更新 docker-compose.yml 中的镜像版本号
# image: jeje_webos:v新版本

# 5. 启动新版本
sudo docker compose up -d
```

## 🗑️ 完全卸载 (重装指南)

如需彻底清除所有数据重新安装：

```bash
# 1. 停止并删除容器及数据卷
sudo docker compose down -v

# 2. 删除本地存储目录
rm -rf ./storage ./logs

# 3. 删除镜像（可选）
sudo docker rmi jeje_webos:v2.5.35

# 4. 清理 Docker 缓存（可选）
sudo docker system prune -f
```

> ⚠️ **警告**: 以上操作将删除所有用户数据，请确保已备份重要文件！
